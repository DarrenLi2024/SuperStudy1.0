## 二、核心协作原则

### 2.1 原子化任务原则
- 每个任务必须是**独立、可验证、无歧义**的原子单元
- 任务之间通过**明确的依赖关系**连接
- 每个任务必须包含：输入依赖、输出产物、验收标准、审计检查点

### 2.2 状态追踪原则
- 所有任务必须标记状态：`pending`（待开始）/ `in_progress`（进行中）/ `done`（完成）/ `audited`（已审计）/ `blocked`（阻塞）
- 每次交接必须更新执行日志
- 状态变更必须由当前执行AI标记

### 2.3 上下文传递原则
- **零上下文启动**：任何AI接手任何任务时，仅需阅读《AI快捷执行手册》即可开始工作
- **状态同步**：完成任务后必须更新任务状态和执行日志
- **变更记录**：修改任何文件必须记录变更内容和原因

### 2.4 冲突解决原则
- **文件锁定**：同时只能有一个AI修改同一个文件
- **版本控制**：使用Git进行版本管理，每次提交前必须pull最新代码
- **冲突处理**：遇到冲突时，暂停任务并在日志中记录，等待人工介入

## 三、角色定义与分工策略

### 3.1 AI角色定义

| **Cursor** | 前端开发、UI组件、交互实现 | 前端页面开发、组件封装、动效实现 | 高优先级前端任务 |

| **WorkBuddy** | 自动化脚本、运维部署、监控 | 自动化运维脚本、部署配置、监控告警 | 低优先级运维任务 |

### 3.2 任务分配策略

#### 策略A：按模块分配（推荐）

| 模块 | 负责AI | 任务范围 |
|-----|--------|---------|
| AI大模型中枢层 | 扣子 | 高考数据抓取、院校聚合、题库生成、学情分析、激励文案 |
| 后端业务服务 | Claude Code | 用户权限、位次换算、自适应学习、段位激励、定时任务 |
| 前端页面 | Cursor | 学生端5页、家长端、管理后台、登录页 |
| 测试与优化 | Codex | 单元测试、集成测试、性能优化、Bug修复 |
| 部署运维 | WorkBuddy | Docker部署、备份脚本、监控告警、CI/CD |
| 全栈集成 | Trae | 接口联调、端到端测试、问题排查 |

#### 策略B：按迭代分配

| 迭代阶段 | 负责AI | 任务范围 |
|---------|--------|---------|
| 一期MVP | Claude Code + Cursor | 登录、档案、模考、首页、管理后台 |
| 二期AI闭环 | 扣子 + Codex | LLM对接、题库生成、错题补强、复盘报告 |
| 三期优化 | WorkBuddy + Trae | 性能优化、运维监控、多学生管理 |

### 3.3 建议采用策略
**推荐采用策略A（按模块分配）**，理由：
1. 每个AI专注于自己擅长的领域，质量更高
2. 模块间通过API接口解耦，减少冲突
3. 便于并行开发，缩短整体周期

## 四、任务卡片标准格式

### 4.1 任务卡片结构

```yaml
任务ID: TASK-XX
任务名称: [简短描述]
负责AI: [分配的AI工具]
当前状态: pending/in_progress/done/audited/blocked
优先级: high/medium/low
依赖任务: [TASK-YY, TASK-ZZ]
输入依赖:
  - [文件路径/接口URL]
  - [文件路径/接口URL]
输出产物:
  - [文件路径/产物名称]
  - [文件路径/产物名称]
验收标准:
  - [可验证的验收条件1]
  - [可验证的验收条件2]
审计检查点:
  - [审计项1]
  - [审计项2]
执行日志:
  - [时间戳] [AI工具] 开始执行任务
  - [时间戳] [AI工具] 完成任务，状态更新为done
```

### 4.2 状态流转规则

```
pending → in_progress → done → audited
              ↑              │
              └──────────────┘
                   blocked ←
```

- **pending** → **in_progress**：AI开始执行时标记
- **in_progress** → **done**：AI完成任务并通过自测时标记
- **done** → **audited**：下一任AI或人工审查通过后标记
- **in_progress** → **blocked**：遇到阻塞问题无法继续时标记

## 五、交接协议

### 5.1 任务交接流程

```
1. 当前AI完成任务，状态更新为done
2. 当前AI记录执行日志（修改的文件、遇到的问题、解决方案）
3. 当前AI标记下一任务为in_progress（如果有）
4. 下一任AI读取执行手册，查看任务状态和执行日志
5. 下一任AI验证上一任务的输出产物
6. 下一任AI开始执行新任务
```

### 5.2 执行日志标准格式

```
[时间戳] [AI工具] [操作类型] [详情]
```

**操作类型说明**：
- `START`：开始执行任务
- `CREATE`：创建新文件
- `MODIFY`：修改现有文件
- `DELETE`：删除文件
- `DEPLOY`：部署代码
- `TEST`：执行测试
- `BLOCKED`：任务阻塞
- `RESOLVE`：解决阻塞问题
- `COMPLETE`：完成任务
- `AUDIT`：审计通过

### 5.3 交接检查清单

**交接前（当前AI）**：
- [ ] 任务状态已更新为done
- [ ] 执行日志已记录
- [ ] 输出产物已验证
- [ ] 测试用例已通过（如适用）
- [ ] 代码已提交到Git

**交接后（下一AI）**：
- [ ] 阅读执行手册和执行日志
- [ ] 验证上一任务的输出产物
- [ ] 检查Git最新提交
- [ ] 确认环境配置正确
- [ ] 开始执行新任务

## 六、冲突解决机制

### 6.1 文件冲突处理

**场景1：同一文件被两个AI同时修改**
- 解决方案：使用Git版本控制，先pull再push
- 如果冲突：暂停任务，记录冲突详情，等待人工介入

**场景2：接口定义不一致**
- 解决方案：以API接口文档为准
- 如果文档与实现不一致：更新文档并通知相关AI

**场景3：数据库schema变更冲突**
- 解决方案：使用Flyway/Liquibase进行数据库迁移管理
- 每次schema变更必须生成迁移脚本

### 6.2 优先级冲突处理

**场景：两个任务同时需要同一个资源**
- 解决方案：按照任务优先级排序
- 高优先级任务优先执行
- 低优先级任务等待资源释放

### 6.3 质量冲突处理

**场景：AI输出质量不达标**
- 解决方案：设立质量门禁
- 验收标准未通过的任务必须重新执行
- 严重质量问题需人工介入评估

## 七、质量保障体系

### 7.1 代码审查清单

**后端代码审查**：
- [ ] 代码风格符合阿里巴巴Java开发手册
- [ ] 接口返回格式符合统一规范
- [ ] 异常处理完善，无空指针风险
- [ ] 数据库查询使用索引，无慢查询
- [ ] 权限校验完整，无越权访问风险
- [ ] 关键业务逻辑有单元测试

**前端代码审查**：
- [ ] 代码风格符合Vue3官方风格指南
- [ ] 组件封装合理，无冗余代码
- [ ] 响应式设计适配移动端
- [ ] API调用有错误处理和加载状态
- [ ] 权限路由控制完整
- [ ] 动效流畅，无卡顿

**AI模块审查**：
- [ ] LLM调用有重试机制和降级方案
- [ ] AI输出有质量校验和过滤
- [ ] Prompt工程合理，无歧义
- [ ] API密钥安全存储，不硬编码

### 7.2 AI输出质量门禁

| 门禁项 | 标准 | 检查方式 |
|-------|------|---------|
| 代码可编译 | 编译通过无错误 | CI自动检查 |
| 测试通过率 | 单元测试覆盖率≥80% | CI自动检查 |
| 接口规范 | 返回格式统一，错误码一致 | API文档校验 |
| 安全合规 | 无硬编码密钥，无SQL注入风险 | 静态代码分析 |
| 性能要求 | 接口响应<2秒（AI接口除外） | 性能测试 |

### 7.3 人工抽检节点

| 节点 | 时机 | 抽检内容 |
|-----|------|---------|
| 一期里程碑 | MVP完成后 | 核心功能完整性、用户体验 |
| 二期里程碑 | AI闭环完成后 | AI自动化能力、数据质量 |
| 三期里程碑 | 全部完成后 | 系统稳定性、性能指标 |
| 关键接口 | 每个接口开发完成 | 接口正确性、安全性 |
