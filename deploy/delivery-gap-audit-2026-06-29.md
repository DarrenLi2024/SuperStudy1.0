## 二、P0 阻塞问题

### P0-1：前端登录 Mock 可绕过真实鉴权（已整改）

**设计要求：** 三角色权限强隔离，JWT 登录，单端登录限制。  
**整改结果：** `frontend/src/views/login/index.vue` 已删除本地 Token 生成和任意账号登录路径；登录失败只提示错误。登录成功后保存后端返回的真实 `studentId`。  
**剩余风险：** 仍需后端集成测试覆盖 Redis Token 二次校验和单端登录限制。  
**结论：** 前端绕权路径已移除，需后端测试验证闭环。

### P0-2：资源级权限校验不完整（已整改，待测试）

**设计要求：** 学生只能看自己，家长只读绑定学生，管理员仅账号管理和系统监控。  
**整改结果：**
- 新增 `StudentAccessService` 统一校验学生本人、绑定家长和管理员访问边界。
- `GrowthController`、`LearningController`、`ExamController`、`QuestionController`、`ParentController` 已补资源级读取校验。
- 写入类接口不再信任路径 `studentId`，改为使用当前登录用户绑定的学生档案。
**剩余风险：** 当前环境无法运行后端集成测试，仍需在 JDK/Maven 可用后验证 403/401 行为。  
**结论：** 代码层红线已补，测试验证未完成。

### P0-3：AI大模型中枢仍未形成真实闭环

**设计要求：** LLM实时抓取、生成、分析、报告，全部AI驱动。  
**当前现状：**
- `DefaultLlmClient` 有远程调用入口，但默认配置为 `local`，无真实模型配置。
- `localFallback()` 返回 `{}`，大量服务随后进入本地规则生成。
- `GaokaoDataServiceImpl.syncScoreRankData()` 根据公式估算位次，不抓取教育考试院数据。
- `CollegeServiceImpl.refreshColleges()` 只插入少量固定种子院校，不抓取官方LOGO和院校素材。
**影响：** 项目仍主要是“规则+降级”，不满足 AI Native 核心定位。  
**结论：** 真实LLM与数据源未接通前，不能声称 AI 全自动。

### P0-4：后端无法在当前环境完成编译/测试验证

**设计要求：** 单元测试覆盖率≥80%，集成测试通过。  
**当前现状：** 当前机器缺少 Java Runtime，`java -version` 失败；`mvn` 不可用。  
**影响：** 后端新增代码存在未验证的编译风险、SQL兼容风险、Bean装配风险。  
**结论：** 必须补齐 JDK/Maven 或 Maven Wrapper 后重新跑全量测试。

## 三、P1 功能差距

### P1-1：模考中心仍是前端硬编码试题（已整改）

**设计要求：** AI自动生成试卷，考完自动出分、自动换算位次、自动更新段位。  
**整改结果：** 模考中心已调用 `/api/v1/question/training/{studentId}` 获取训练题，提交后调用后端考试提交接口并刷新历史记录。  
**剩余风险：** 题目质量依赖真实 LLM 和题库数据源配置。

### P1-2：任务完成率未真正回写（已整改）

**设计要求：** 今日任务完成进度实时刷新。  
**整改结果：** 学生首页勾选任务已调用后端完成接口，失败时恢复 UI 状态并提示错误。  
**剩余风险：** 需要后端联调验证完成率聚合到家长端周完成率。

### P1-3：院校“仅稳妥对标”数据基础不足

**设计要求：** 位次±5000匹配稳妥院校。  
**当前现状：** 表结构已增加 `min_rank/max_rank`，但初始化数据仍未填这些字段；真实匹配会回退到批次随机。  
**影响：** “仅稳妥院校”仍未真正按位次落地。

### P1-4：一分一段数据不满足近三年要求

**设计要求：** 近三年等效位次，支持物理/历史。  
**当前现状：** 初始化 SQL 仍只有 2025 河南物理/历史模拟数据；服务缺真实近三年抓取。  
**影响：** 位次换算精度和适用范围不足。

### P1-5：家长端是固定/Mock概况（已整改）

**设计要求：** 家长端只读查看孩子段位、分差、周点评、完成率、模考趋势。  
**整改结果：** `ParentController` 已聚合成长卡片、今日任务、成长数据和模考记录；前端失败时不再使用 Mock 兜底。  
**剩余风险：** 需要以绑定家长账号做只读权限集成测试。

### P1-6：管理员后台存在大量 Mock 降级（已整改）

**设计要求：** 管理员仅账号管理、运行日志、自动备份状态、AI参数基础配置。  
**整改结果：** 管理后台接口失败不再提示成功；监控数据改为读取运行时内存、磁盘、Redis、备份状态和 AI 配置状态。  
**剩余风险：** 数据库健康状态仍以应用运行状态为基础展示，建议后续增加真实连接池探针。

## 七、推荐修复优先级

### 第一优先级：后端验证与生产数据源（1-3天）
- 安装 JDK 17 与 Maven，执行后端编译、单元测试和集成测试。
- 接入真实官方一分一段数据源或可靠数据服务。
- 接入院校录取位次、院校素材、LOGO 的合规公开数据源。
- 配置生产 `JWT_SECRET` 与真实 `AI_LLM_*`。

### 第二优先级：真实AI质量门禁（3-7天）
- 题库生成加入 JSON Schema 校验、质量评分、失败重试。
- 学情报告、激励文案、错因分析加入输出安全和长度约束。
- 增加 AI 调用审计日志和缓存命中率监控。

### 第三优先级：体验增强（2-4天）
- 自动生成考后诊断报告。
- 增加段位升级弹窗和动效。
- 管理后台补系统日志列表、备份日志详情和 AI 调用明细。

### 第四优先级：验证与上线门禁（2-4天）
- 安装 JDK/Maven 或补 Maven Wrapper。
- 后端全量 `mvn test`。
- 前端构建和关键页面走查。
- 补权限隔离集成测试。
- 补红线审计用例。

## 八、MVP交付结论

当前版本：**准生产联调版，暂不建议直接生产发布。**

主要原因：
1. 真实高考位次、院校录取位次和院校素材数据源未接通。
2. 生产 LLM Provider 与密钥未配置，AI质量未完成真实验证。
3. 后端没有完成本地编译和测试验证。
4. 数据库迁移、Redis 定时日志、备份恢复流程尚未在真实环境演练。
